# Notes - LeetCode 3509: Maximum Product of Subsequences With an Alternating Sum Equal to K (C++ proposals)

## Problem Summary

Choose a non-empty subsequence of `nums` whose alternating sum is exactly `k`, where
the signs depend on the chosen subsequence positions: `+ - + - ...`. Among all valid
subsequences, return the maximum product that does not exceed `limit`; return `-1` if
none exists.

## Proposals in This Folder (C++)

This folder holds the **C++** recommended proposal. It is also the runtime extreme, so
there is no separate `solution-runtime.cpp`: that file would be coincident with
`solution.cpp`.

- **Recommended + speed extreme (`solution.cpp`) - fast + lean:** compressed
  product-indexed bitset DP. Each positive product row stores reachable alternating
  sums for odd/even subsequence length. Zero-product subsequences are tracked with a
  separate reachability DP.

The minimum-memory champion is implemented in C - see [See Also](#see-also).

## Language Choice (C++)

The hot path is a small number of fixed-width bitset shifts. C++ gives native execution,
inlined `std::bitset` word operations, and compact contiguous vectors without the
allocation overhead of hash-set based DP. That makes it the best raw-speed choice while
still keeping memory low.

## Constraints

- `1 <= nums.length <= 150`
- `0 <= nums[i] <= 12`
- `-1e5 <= k <= 1e5`
- `1 <= limit <= 5000`

The useful alternating-sum range is much smaller than the input `k` range:
`sum(nums) <= 150 * 12 = 1800`, so `|k| > sum(nums)` is impossible.

Positive products are also highly compressible. Although `limit` can be 5000, products
are built only from factors `1..12`; under 5000 there are at most 394 distinct positive
product values when all factors are available.

## How the Solution Was Derived

For a partially chosen subsequence, the next sign is determined only by the current
length parity:

- even length -> the next chosen value is added and the new length is odd;
- odd length -> the next chosen value is subtracted and the new length is even.

That gives the natural DP state `(product, parity, alternating_sum)` for positive
products not exceeding `limit`.

A zero requires separate handling. If a positive-product prefix already exceeds
`limit`, it is useless for future positive products, but it can still become a valid
product-0 subsequence after choosing a later zero. Therefore the solution also maintains:

- `anyOdd` / `anyEven`: all non-empty subsequences by alternating sum and parity, with
  product ignored;
- `zeroOdd` / `zeroEven`: all non-empty subsequences that contain at least one zero.

## Approach

1. Compute `total = sum(nums)` and immediately reject impossible `k`.
2. Precompute the sorted closure of positive products `<= limit` generated by values
   present in `nums`; map each product to a dense row index.
3. For every positive product row, keep two bitsets of alternating sums: odd length and
   even length.
4. For each number `x`:
   - update the product-agnostic and zero-containing reachability bitsets;
   - if `x > 0`, multiply active product rows by `x` and shift their sum bitsets by
     `+x` or `-x`;
   - add the singleton subsequence `[x]` when `x <= limit`.
5. Scan compressed product rows from largest to smallest. If none matches `k`, return
   `0` only if a zero-product subsequence reaches `k`; otherwise return `-1`.

For `x > 1`, rows are processed in descending product order, so a state created by the
current element is never reused by that same element. For `x == 1`, the product row is
unchanged, so the old odd/even bitsets are copied before toggling parity.

## Top 1% Performance Strategy

- Reject impossible `k` before allocating the DP table.
- Use the real sum bound (`<= 1800`), not the stated `1e5` range.
- Compress product rows to the closure of attainable products (`<= 394` rows instead of
  5001).
- Store alternating sums as bitsets, so each transition is a word-level shift and OR.
- Keep zero-product reachability outside the product DP, avoiding a product-0 row that
  would need special transitions.
- Process product rows in descending order to avoid cloning the whole DP table per item.

## Trade-Offs

- The C++ solution is the best default and the speed extreme: it uses fixed bitsets and
  dense product indices for excellent constant factors.
- The C memory proposal uses exact-width manual bitsets and C allocation to reduce
  reported memory further, at the cost of more verbose and less ergonomic code.

## Edge Cases

- `k` outside `[-sum(nums), sum(nums)]` -> `-1`.
- All zeros and `k == 0` -> `0`.
- Positive products over `limit` are ignored, but zero-product reachability still remains
  correct through the product-agnostic DP.
- `x == 1` needs special handling because multiplying by one keeps the same product row.
