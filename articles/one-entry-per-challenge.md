title: One entry per challenge — deduplicating a multi-language manifest
date: 2026-06-09
summary: A counting bug postmortem. The site said 38 solved when the truth was 32, because the manifest's unit of count silently changed the day one challenge was solved in two languages.
tags: postmortem, build pipeline, data modeling

This site is fully data-driven: every page derives from a single
`platforms/manifest.json`, generated by walking each challenge's
`metadata.json`. That design has one obligation that is easy to state and
easy to get wrong: **everyone has to agree on what "one challenge" means.**

For a while, nobody did — including the generator.

## The symptom

The homepage hero claimed more solved challenges than the per-platform stats
page did (38 against 32 at the time it was caught). The LeetCode index showed
duplicate rows: the same problem id appearing twice, identical except for the
language column. Both pages read the same manifest, so the bug had to be
upstream of both.

## The cause

The repository's layout is *one folder per (challenge, language)*. A problem
solved in C++ and revisited in C as a didactic exercise legitimately lives in
two folders, each with its own `metadata.json`:

```text
platforms/leetcode/cpp/0217-contains-duplicate/   <- recommended + runtime
platforms/leetcode/c/0217-contains-duplicate/     <- memory extreme
```

The manifest generator walked every `metadata.json` and emitted one manifest
entry per file. So the manifest's implicit unit of count was the *language
solution*, while every consumer believed it was the *challenge*. The canonical
model — already enforced by the stats generator — is explicit: a challenge is
one distinct `(platform, id)`; multi-language challenges count once everywhere
except the per-language breakdown.

The drift was invisible at first because the two units coincide while every
challenge has exactly one language. The first re-solve in a second language
broke the equivalence, and every counter downstream inflated at once.

## The fix: dedupe at the producer

One consumer (the pattern/topic lens) had already grown a local workaround — a
client-side dedup before rendering. That is the classic wrong place: fixing
one reader leaves every other reader wrong, and the workaround itself hides
the upstream defect.

The fix moved the merge into the generator. `mergeChallenges()` collapses the
per-language summaries into one entry per `(platform, id)`:

- `languages[]` lists every language the challenge was solved in, sorted.
- `topics` and `patterns` are unioned across the language folders.
- The **primary** path and complexity come from the variant that ships a plain
  `solution.<ext>` — the recommended proposal — never from a `-runtime` or
  `-memory` extreme.
- `dateSolved` keeps the earliest date, so the learning timeline reflects the
  first solve, not the latest revisit.
- The input is sorted `(platform, id, language)` before merging, so the output
  is deterministic and the generator stays idempotent — running it twice
  produces byte-identical files, which keeps the pre-commit hook honest.

With the manifest correct, the lens's local dedup was deleted, not kept "just
in case": a workaround that survives its root cause becomes dead logic that
will mislead the next reader.

## What this generalizes to

- **Name your unit of count.** "37 challenges" and "44 language solutions
  across 44 metadata files" are both true today — the generator now prints
  both, so the ambiguity is impossible to reintroduce silently.
- **Aggregate at the producer, not the consumers.** N readers of one index
  should never each re-derive the data model.
- **Watch for coincidental invariants.** The bug shipped green because two
  different units happened to be numerically equal. The first counterexample
  arrived months later, far from the code that made the assumption.
